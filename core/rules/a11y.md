# Accessibility (a11y)

Mandatory checks for every interactive surface.

## Universal Principles

- Every interactive element has a programmatic accessible name
- Disabled state explains WHY (programmatic, not visual-only)
- Errors are programmatically linked to their fields
- Dynamic state changes are perceivable (announced, focused, or audible)
- Color/icon is never the sole signifier of meaning
- Keyboard-only users can reach and operate everything
- Focus is managed on dynamic content (open → trap, close → return)
- Heading hierarchy matches content structure
- Time-based or motion content can be paused / reduced

## Severity Ranking

| Issue | Severity |
|-------|----------|
| Missing accessible name on interactive control | HIGH |
| Keyboard trap (focus can't escape) | HIGH |
| Form submission blocks screen reader from reaching the form | HIGH |
| Disabled state without programmatic reason | MEDIUM |
| Error message not linked to its field | MEDIUM |
| Color-only status indicator | MEDIUM |
| Dynamic content change without announcement | MEDIUM |
| Heading-level skip | LOW |
| Missing focus-visible outline | LOW |

## Common Misses (from prior reviews)

- Disabled submit button with no programmatic explanation of why (e.g. "≥ 3 chars" rule)
- Icon-only buttons lacking accessible name
- Inline form error rendered far from its field, not programmatically linked
- Hidden helper inputs (e.g., file input behind a styled button) appearing as duplicate same-labeled elements
- Dialog dismiss-on-ESC/backdrop while submit is in-flight
- Toast/banner appears with no aria-live announcement
- Loading state without aria-busy or visible spinner

## Required Tooling

Each profile must list a11y tools in its `context.md`. Examples:
- Automated audit (axe-core, Lighthouse, pa11y)
- Component-level a11y assertions in unit tests
- Manual keyboard walkthrough as part of visual verification

## Pre-Commit Requirements

- [ ] Every interactive element reachable by keyboard
- [ ] Every form field has a programmatic name
- [ ] Every error is linked to its field
- [ ] No color-only signifiers
- [ ] Dynamic state changes are announced
- [ ] Dialogs trap focus and return it on close
